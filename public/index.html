Fraud Detection

Automated detection of non-human and fraudulent traffic patterns

Fraud Risk by Vendor
Detection Heuristics

Bot Signature Detection

Identifies known bot user-agent strings (Googlebot, python-requests, curl, etc.) in incoming traffic.

  • Weight: 25% of fraud score
  • Signal: knownBotSignature

Click Velocity Analysis

Flags clicks with redirect latency < 100ms. Real users on real devices have measurable network hop time.

  • Weight: 20% of fraud score
  • Signal: suspiciouslyFastRate

IP Concentration

Measures how concentrated traffic is across IPs. Botnet/proxy abuse funnels through few IPs.

  • Weight: 15% of fraud score
  • Signal: ipConcentrationRate

IP Diversity Ratio

Unique IPs ÷ total clicks. Low diversity indicates traffic from a small pool of machines.

  • Weight: 15% of fraud score
  • Signal: ipDiversityRatio

Temporal Uniformity

Real traffic peaks during business hours. Perfectly uniform hourly distribution is suspicious.

  • Weight: 10% of fraud score
  • Signal: temporalEntropy

Click Timing Variance

Humans click at irregular intervals (high CV). Bots click with clockwork regularity (low CV).

  • Weight: 10% of fraud score
  • Signal: clickTimingVariance

How Experiments Work

Non-Disruptive Design

Only 10% of campaign traffic is diverted to experiments. The remaining 90% flows normally, ensuring live campaigns are never disrupted.

Sequential Testing

Results are evaluated after each batch. If statistical significance (p < 0.05) is reached early, the experiment auto-completes — saving budget.

Statistical Rigor

Uses two-proportion z-tests with Wilson confidence intervals. Each vendor's engagement rate is compared against the pooled average.

Auto-Scheduling

The engine prioritizes vendors with lowest confidence scores, ensuring under-tested vendors get evaluated first while respecting concurrency limits.

Thompson Sampling — Traffic Allocation

Traffic allocation uses a multi-armed bandit algorithm that balances exploration vs exploitation:

  • Explore: uncertain vendors get sampled from wide Beta posteriors — they might be good
  • Exploit: known-good vendors have narrow posteriors centered high — they get more traffic
  • Adapt: when vendor quality drops, the posterior shifts and allocation decreases automatically
  • No manual tuning: the algorithm is provably near-optimal for regret minimization
Bayesian Scoring Model

Each vendor's quality is modeled as a Beta(α, β) distribution:

  • Prior: Beta(2, 2) — weak prior expressing initial ignorance
  • Update: α += successes, β += failures (from engagement proxy)
  • Decay: 0.95× per cycle — older data fades, enabling adaptation to vendor changes
  • Confidence: log₁₀(sampleSize)/3 — naturally increases with more data
  • Ranking: compositeScore × confidence — prevents small-sample vendors from ranking too high

Event & Data Handling

How the system processes traffic events and handles edge cases

Event Lifecycle

Each incoming click goes through a structured processing pipeline:

  • Ingestion: TrafficEvent captured with full metadata at redirect layer
  • Enrichment: IP lookup, device detection, geo resolution, bot signature check
  • Experiment assignment: probabilistic assignment to running experiments (10% allocation)
  • Signal aggregation: events are batched into time-windowed QualitySignals per vendor
  • Score update: Bayesian posterior is updated with new evidence
  • Redirect: visitor is forwarded to the landing page (VDP) with no delay
Edge Cases Handled
  • New vendor (cold start): starts with Beta(2,2) prior — wide uncertainty, gets exploration traffic via Thompson Sampling
  • Vendor behavior change: decay factor (0.95) ensures old data fades; score adapts within 3-5 cycles
  • Burst traffic: temporal entropy detection flags sudden concentrated bursts as anomalous
  • IP spoofing/rotation: IP diversity ratio + rapid-fire cluster detection catches coordinated bot pools
  • Zero-click window: empty signal windows produce neutral scores — no false penalization
  • Small sample sizes: Wilson confidence intervals + Bayesian confidence prevent premature conclusions
  • Concurrent experiments: max concurrency limit prevents over-diversion of live traffic
  • Geo mismatch: automotive vertical relies on local geo — out-of-area traffic is heavily penalized in scoring
Data Volume Strategy

Windowed Aggregation

Raw events are aggregated into time-windowed signals (hourly/daily). This reduces storage from O(events) to O(vendors × windows) while preserving all quality information needed for scoring.

  • Raw events: retained 7 days
  • Quality signals: retained 90 days
  • Vendor scores: retained indefinitely
Failure Modes

Graceful Degradation

The system is designed to fail safely at every layer:

  • Missing geo data → neutral geo score (not penalized)
  • Scoring service down → traffic continues via last-known allocation weights
  • Experiment engine down → all traffic uses production routing
  • Vendor goes offline → auto-detected via zero-click window
Production Scale Path

From POC to Production

The in-memory store is backed by a clean interface. Production swap:

  • Events → TimescaleDB (time-series optimized)
  • Scores → PostgreSQL (relational)
  • Real-time → Kafka + Flink streaming
  • API → Horizontal scaling behind load balancer
Traffic Event Schema
interface TrafficEvent {
  id:                    string        // unique event identifier
  vendorId:              string        // which vendor sent this traffic
  campaignId:            string        // which campaign it's for
  timestamp:             Date          // when the click occurred
  experimentId?:         string        // assigned experiment (if any)
  ip:                    string        // visitor IP address
  userAgent:             string        // browser/bot user-agent
  deviceType:            DeviceType    // mobile | desktop | tablet | unknown
  geo:                   GeoInfo       // country, region, city, zip
  referrer:              string        // HTTP referrer header
  redirectLatencyMs:     number        // ms between click and redirect
  knownBotSignature:     boolean       // matched known bot UA pattern
  repeatIp:              boolean       // IP seen before in recent window
  timeSinceLastClickFromIp: number|null // seconds since last click from same IP
}
