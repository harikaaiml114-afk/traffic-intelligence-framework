Thompson Sampling

Each vendor has a Beta(a,b) posterior. To pick where the next click goes, we sample from every vendor's distribution and route to whoever draws highest. Uncertain vendors occasionally draw high values (exploration), while consistently good ones draw high most of the time (exploitation). No tuning knobs needed — it's self-balancing.

Bayesian Decay

Old observations get multiplied by 0.95 each scoring cycle. This means a vendor that was good 2 weeks ago but tanked recently won't coast on stale data — the model catches up in 3-5 cycles. We picked 0.95 as a balance between stability and responsiveness.

Event Handling

What happens to each click
  1. Click hits our redirect endpoint — we grab IP, UA, device, geo, timestamp
  2. Check UA against bot list, measure redirect latency
  3. Roll dice: does this click go into an active experiment? (10% chance)
  4. Store the TrafficEvent, forward the visitor to the landing page
  5. Periodically: batch events into QualitySignals per vendor per hour
  6. Run scoring: update that vendor's Beta posterior
Edge cases
  • New vendor — starts at Beta(2,2), gets extra exploration via Thompson Sampling
  • Vendor quality drops — decay factor means score adapts in ~4 cycles
  • Traffic burst — temporal entropy spikes, flagged as anomalous
  • IP rotation — rapid-fire cluster detection catches coordinated bots
  • No clicks in window — neutral signal, no false penalty
  • Tiny sample — Wilson intervals + wide posterior prevent overreaction
  • Too many experiments — hard cap on concurrent experiments per campaign

Retention

Raw events: 7d. Signals: 90d. Scores: indefinite. Production would use TimescaleDB for events, Postgres for the rest.

Failures

Scoring down? Last-known weights keep serving. Experiment engine down? All traffic uses prod routing. Vendor offline? Detected via zero-click window.

Scaling

Current: in-memory, single process. Prod path: Kafka ingest, Flink for streaming aggregation, horizontal API scaling behind LB.
